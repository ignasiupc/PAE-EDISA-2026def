1 ¿Qué es el Drone GCS?

El Drone GCS (Ground Control Station) es una aplicación de escritorio que actúa como panel de control en tiempo real para un dron hexarotor. Desde una sola ventana, el operador puede ver el estado del dron, el vídeo de las cámaras, el mapa de navegación, los datos de los sensores y los resultados del procesamiento de imágenes (códigos de barras, volumetría).

La aplicación corre en Windows/Linux/Mac mediante Electron (el mismo motor que VSCode) y se comunica con el dron a través de la red WiFi local, utilizando el protocolo ROS2 (Robot Operating System) como capa de mensajería.

¿Para quién es este documento? Está pensado para cualquier miembro del equipo con conocimientos básicos de informática. No se necesita experiencia previa en robótica ni en sistemas embebidos para entender la arquitectura y el funcionamiento.

3 Interfaz de Usuario

La aplicación tiene cuatro pestañas, seleccionables en la barra superior. En la misma barra aparece el estado de conexión (punto verde/rojo), la URL del servidor ROS y las insignias de estado del dron (armado, modo de vuelo, GPS).

Dashboard

Vista principal de vuelo, dividida en tres columnas:

  • Izquierda: velocímetro animado, altitud, heading, batería, modo/armado, coordenadas GPS.
  • Centro: feed de vídeo de la cámara frontal (MJPEG directo desde la Pi).
  • Derecha: mapa de navegación con trayectoria GPS + tabla de códigos de barras escaneados.

El diagrama hexarotor en la zona inferior de la cámara muestra en tiempo real el empuje (%) de cada uno de los 6 motores con arcos de color.

Navigation

Instrumentos de pilotaje clásicos:

  • Horizonte artificial — muestra roll y pitch con cielo/tierra animados.
  • Brújula — aguja roja apuntando al norte magnético.
  • Panel de datos — velocidad, velocidad vertical, altitud, ángulos de Euler, GPS completo, temporizador de vuelo.
  • Gráficas en tiempo real — altitud, velocidad y velocidad vertical (últimos 120 puntos).
SLAM

Mapa de ocupación generado por SLAM Toolbox + overlay del LiDAR:

  • Mapa de celdas ocupadas/libres (/map) renderizado célula a célula.
  • Nube de puntos del LiDAR (/scan) superpuesta en la pose actual del robot.
  • Panel lateral con pose X/Y/θ, dimensiones del mapa, resolución, rangos del LiDAR y contador de loop closures.
Image Processing

Análisis visual de las dos cámaras:

  • Cam 1 (frontal) y Cam 2 (inferior) en tiempo real via ROS CompressedImage.
  • Tabla de detecciones (códigos de barras, objetos) con etiqueta, ID, confianza y timestamp.
  • Log de inserciones en base de datos.
  • Panel de volumetría: dimensiones y volumen del último objeto detectado, historial.

4 Flujo de Datos en Tiempo Real

Todos los datos del dron llegan a la aplicación como topics ROS2, que son como "canales de radio" con nombre donde cada sensor/módulo publica sus medidas. La app se suscribe a estos canales y actualiza la interfaz al instante.

1
Sensor → PX4/MAVROS
El autopilot publica datos de vuelo (velocidad, altitud, batería, actitud, GPS) como topics /mavros/... en ROS2.
2
ROS2 → rosbridge → WebSocket
rosbridge escucha en el puerto 9090 de la Raspberry Pi y convierte cada mensaje ROS2 a JSON que viaja por la red WiFi.
3
WebSocket → roslib.js → estado S
La librería roslib.js en la app recibe el JSON, deserializa el mensaje y llama al callback correspondiente, que actualiza el objeto de estado global S.
4
Estado S → Canvas 2D → pantalla
Un bucle de animación (requestAnimationFrame, ~60 fps) lee S y redibuja todos los instrumentos: velocímetro, batería, mapa, SLAM, gráficas, diagrama de motores.
5
Cámara (ruta paralela)
El vídeo toma dos caminos independientes: MJPEG directo vía HTTP (panel principal, baja latencia) y topic ROS CompressedImage via rosbridge (pestaña Image Processing, sincronizado con ROS).

5 Sistema de Cámaras

El dron lleva dos cámaras Pi Camera Module: una frontal (orientada hacia adelante, seguimiento de obstáculos y navegación) y una inferior (apuntando hacia abajo, detección de códigos de barras y volumetría).

Publicación en la Raspberry Pi

El script camera_publisher.py usa la librería picamera2 (específica de Pi Camera Module) para capturar fotogramas. Cada frame se comprime a JPEG y se publica como mensaje ROS2 de tipo sensor_msgs/CompressedImage.

¿Por qué picamera2 y no OpenCV VideoCapture? Las cámaras CSI de Raspberry Pi (IMX708) usan el stack libcamera, incompatible con la interfaz V4L2 estándar que utiliza OpenCV. picamera2 es la librería oficial para acceder a estas cámaras correctamente en Raspberry Pi OS. 
# Lanzar cámara frontal (dispositivo 0)
python3 camera_publisher.py --ros-args \
  -p device:=0 -p fps:=30 -p width:=640 -p height:=480

# Lanzar cámara inferior (dispositivo 1)
python3 camera_publisher.py --ros-args \
  -p device:=1 -p topic:=/camera/down/image_raw/compressed

6 Conexión App ↔ Raspberry Pi

Pasos para conectar

1
Raspberry Pi y PC en la misma red WiFi
Conectar ambos dispositivos al mismo punto de acceso. La Pi tiene IP estática o dinámica (típicamente en el rango 10.4.91.x o 192.168.x.x).
2
Lanzar servicios en la Pi 
./start_gcs.sh    # inicia rosbridge + camera_publisher
3
Abrir la app y conectar
Clicar en la URL de la barra superior → aparece el popover de conexión. Introducir ws://<IP de la Pi>:9090 o usar el botón "Scan network for Raspberry Pi" para detectarla automáticamente.
4
Punto verde = conectado
El punto de la barra superior se pone verde. La URL se guarda en memoria local para la próxima sesión.

Escáner de red automático

Si no se conoce la IP, el botón de escaneo prueba todas las IPs de la subred /24 (254 hosts) en paralelo con 40 conexiones simultáneas y timeout de 400 ms. Detecta la IP local del PC vía WebRTC para determinar qué subred escanear. El escaneo completo tarda aproximadamente 3–4 segundos.

7 Scripts de la Raspberry Pi

ArchivoFunciónPuerto / Topic
start_gcs.sh Script maestro: lanza rosbridge + camera_publisher. Gestiona PIDs y limpieza al salir con Ctrl+C. —
camera_publisher.py Nodo ROS2. Lee picamera2, publica JPEG a 30 fps. Parámetros: device, topic, fps, width, height, jpeg_quality. /camera/forward/… /camera/down/…
mjpeg_server.py Servidor Flask MJPEG. No requiere ROS. Útil para streaming de baja latencia en el dashboard. :8080/cam1
Nota sobre rosbridge El paquete ros-jazzy-rosbridge-suite no está disponible vía apt en mayo 2026. Se compiló desde el código fuente (Release 2.3.0) en ~/rosbridge_ws en la Pi. Para activarlo: source ~/rosbridge_ws/install/setup.bash antes de lanzar.

8 Instalación y Arranque

En el PC (app)

# Requisito: Node.js ≥ 20 instalado
git clone https://github.com/clarajorba/PAE-EDISA-2026.git
cd PAE-EDISA-2026/WP2/gcs
npm install
npm start          # abre la ventana GCS

# Para generar instalador (.exe / .AppImage):
npm run build      # → dist/Drone GCS Setup 1.0.0.exe

En la Raspberry Pi

# Requisitos: ROS2 Jazzy, picamera2, flask, opencv-python
# 1. Activar ROS2 + rosbridge
source /opt/ros/jazzy/setup.bash
source ~/rosbridge_ws/install/setup.bash

# 2. Lanzar todo
cd ~/PAE-EDISA-2026/WP2/gcs/raspberry
./start_gcs.sh    # rosbridge :9090 + cámara

# Opcionalmente, lanzar el servidor MJPEG en paralelo:
python3 mjpeg_server.py   # :8080/cam1
