## Build & Test

All commands run from the workspace root `~/meu_projecte_ros/ros2_ws/`:

```bash
# Build this package only
colcon build --packages-select dron_control_sistema

# Run linting tests (flake8 + pep257)
colcon test --packages-select dron_control_sistema
colcon test-result --verbose

# Source the workspace after building
source install/setup.bash
```

## Running the Simulation

Three nodes must be launched in separate terminals (order matters — start simulator first):

```bash
# Terminal 1: physics simulator + live 3D plot
ros2 run dron_control_sistema simulador_dron

# Terminal 2: velocity controller bridge
ros2 run dron_control_sistema mavros_node

# Terminal 3: mission/waypoint sequencer
ros2 run dron_control_sistema cerebro_node
```

`lidar_node` is not part of the normal pipeline — it publishes random poses to `/drone/pose`, which conflicts with `simulador_dron` if both run simultaneously.

## Standalone Analysis Scripts

Run directly with Python (no ROS daemon needed):

```bash
cd dron_control_sistema/dron_control_sistema/

# Monte Carlo analysis → estadistica_mc.png, mapa_decision.png, mision.json
python3 simulacion_montecarlo.py

# Interactive 3D visualization → resultado_interactivo.html
python3 simulacion_interactiva.py
```

`simulacion_montecarlo.py` also regenerates `mision.json`, which `cerebro_node` reads at startup.

## Architecture

The system simulates a drone scanning three warehouse shelves in a **triple-Z pattern** (zigzag across each shelf). All motion is in the XY plane at a fixed Z altitude of 1.0 m.

### Node Communication

```
simulador_dron  →  /drone/pose  →  cerebro_node
                                →  mavros_node

cerebro_node    →  /cerebro/target  →  mavros_node

mavros_node     →  /drone/cmd_vel   →  simulador_dron
```

### Node Responsibilities

| Node | Role |
|---|---|
| `cerebro_node` | Loads `mision.json`, sequences waypoints, publishes next target when drone reaches current one (XY distance < `margen`, default 0.18 m) |
| `mavros_node` | Proportional velocity controller: `cmd_vel = target - current_pos` (raw difference, no scaling) |
| `simulador_dron` | Integrates `cmd_vel` into position (`pos += vel`), publishes pose at 2 Hz, renders live Matplotlib 3D plot |
| `lidar_node` | Unused in pipeline; publishes random poses for isolated testing |

### Message Types

- `/drone/pose` — `std_msgs/Float32MultiArray`: `[x, y, z, yaw]`
- `/cerebro/target` — `std_msgs/Float32MultiArray`: `[x, y, z]`
- `/drone/cmd_vel` — `geometry_msgs/Twist`: linear `{x, y, z}` velocities

Custom ROS message types are not used; the package relies on standard `std_msgs` and `geometry_msgs`.

### Mission Configuration (`mision.json`)

Generated by `simulacion_montecarlo.py`. Contains:
- Dense waypoints at 0.3 m spacing along the triple-Z path (~120+ checkpoints)
- `margen`: arrival tolerance in metres
- `z_fijo`: fixed flight altitude

`cerebro_node` resolves this file relative to the script's own directory (`__file__`).

### Triple-Z Geometry

Three shelves at X = {0, 5, 10}, each 3 m wide × 4 m tall, separated by 2 m gaps. Per shelf: bottom row right → diagonal up-left → middle row left → top row right, then drop and transition to next shelf.

## Code Conventions

- **Linting:** flake8 (line length 99, enforced in tests) and pep257 docstrings.
- **Configuration:** JSON file, not ROS parameters or YAML.
- **Logging:** `self.get_logger().info(...)` only; no `print()` in nodes.
- **Control:** Simple P controller (`K_p = 2.0` in Monte Carlo scripts; direct difference in `mavros_node`).

## Dependencies

Python: `rclpy`, `numpy`, `matplotlib`, `plotly`, `scipy`  
ROS msgs: `std_msgs`, `geometry_msgs`  
Test tools: `ament_flake8`, `ament_pep257`, `pytest`
